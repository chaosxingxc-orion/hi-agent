  并发能力：零

  15 个核心文件中没有一个使用 asyncio、Lock、或任何并发原语。 RunExecutor.execute() 是纯同步 for 循环。要同时跑两个 Run，当前代码会直接竞态崩溃。

  ---
  对"大规模并发使用"的距离判断

  "大规模并发"意味着：
  1. 多个 Run 同时执行，互不干扰
  2. 每个 Run 内的 LLM 调用、Capability 调用都是异步非阻塞
  3. 状态持久化在进程崩溃后可恢复
  4. 外部事件（callback、Human Gate 审批）可以异步到达
  5. 熔断、限流、超时保护到位

  ┌──────────────────────────┬───────────────────────────────────────────────────────────────┬────────────────────────────────────────────────────────┐
  │           能力           │                           当前状态                            │                       需要做什么                       │
  ├──────────────────────────┼───────────────────────────────────────────────────────────────┼────────────────────────────────────────────────────────┤
  │ 多 Run 并发              │ 不可能——共享可变状态、同步阻塞                                │ runner.py 重写为 async + 每 Run 独立状态隔离           │

  ├──────────────────────────┼───────────────────────────────────────────────────────────────┼────────────────────────────────────────────────────────┤

  ---
  给一个清醒的工程计划

  不再按"阶段"组织（那是项目管理视角），改按**"从 spike 到生产需要闯过的 6 道关"**组织：

  关卡 1：async 化（1 周）

  不做这步，后面全部白搭。

  runner.py:     execute() → async execute()
  invoker.py:    invoke() → async invoke() + asyncio.wait_for(timeout)
  emitter.py:    emit() → async emit() + 异步写入队列
  compressor.py: compress() → async compress()（预留 LLM 调用位）
  adapter:       所有 protocol 方法 → async

  每个 Run 在独立的 asyncio Task 中执行，共享状态通过 RunContext 隔离。

  关卡 2：接入真实 kernel（1-2 周）

  从 mock 到真实 durable runtime。

  1. RuntimeAdapter Protocol 扩展到 17 个方法
  2. KernelFacadeAdapter 对接 agent-kernel（先用 LocalFSM substrate）
  3. runner.py 改用 start_run / query_trace_runtime / open_branch 等真实 API
  4. 事件不再写内存 list，而是通过 kernel 的 EventLogStore 持久化
  5. 验证：kill 进程后重启，Run 从 event log 恢复继续

  关卡 3：接入真实 LLM（1 周）

  从内存 mock handler 到真正调 LLM。

  1. Route Engine：规则引擎 + LLM fallback（RT-01 的 V1.5 混合策略）
  2. L1 压缩：Haiku 调用（RT-02 的 prompt B）
  3. Task View 构建：分层读取（L2→L1→L0 fallback）
  4. Capability handler：接入 agent-core 的 LLMClient / ToolClient
  5. 成本记录：每次 LLM 调用记录 input_tokens / output_tokens

  关卡 4：补齐核心安全机制（1 周）

  不做这步，任何 failure 场景都会把系统卡死。

  1. circuit_breaker.py 重写：加入 half_open 状态 + 定时器恢复
  2. invoker.py：加 asyncio.wait_for(timeout) + max_retries + 指数退避
  3. dead_end 检测集成到 runner（不是单独的函数，而是 stage 循环中每次 branch 状态变更后检查）
  4. runner.py 顶层异常保护：任何 stage 执行异常 → 尝试 recovery → 失败则 Run → failed

  关卡 5：StageGraph 驱动（3-5 天）

  从硬编码列表到真正的图驱动执行。

  1. runner.py 的 STAGES 列表 → StageGraph.get_next_stages(current_stage)
  2. Stage 转移基于 StageGraph 的合法转移规则
  3. 回退由 StageGraph 的回退边控制（不是硬编码）
  4. Stage 转移前调用 StageGraph 的形式化验证
  5. 不同 task_family 可以有不同的 StageGraph（通过 TaskFamilyConfig 配置）

  关卡 6：并发 Run 隔离（3-5 天）

  从"一次跑一个 Run"到"同时跑 N 个 Run"。

  1. RunContext：每个 Run 独立的状态容器（dag、stage_summaries、raw_memory）
  2. RunManager：管理多个并发 RunContext，提供 list_runs / cancel_run
  3. Capability 并发：invoker 的 circuit_breaker 按 capability_id 隔离
  4. Event 并发：emitter 按 run_id 分流
  5. 压力测试：同时启动 10 个 quick_task Run，验证无竞态

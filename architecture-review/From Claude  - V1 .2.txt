### 二、V1.2 新增内容的独立评审

V1.2 主动新增了三块重要内容：Runtime Truth Model（6.7）、Idempotency and Side-Effect Contract（6.8）、Failure Taxonomy（6.9）。这三块都不是我上轮要求的，是架构师自主延伸的。逐一评审。

---

**6.7 Runtime Truth Model——高价值增量，但有一个状态缺失**

六组状态机（RunState / StageState / BranchState / ActionState / WaitState / ReviewState）的引入是正确的决策。对于一个 7×24 持久运行的系统，没有显式状态机就没有可调试性。状态转移图也是合理的。

**但 ActionState 缺少一个关键状态：`succeeded`。** 当前定义是 `prepared → dispatched → acknowledged | effect_unknown | failed`。`acknowledged` 暗示外部系统确认收到了请求，但"收到"不等于"成功完成"。在异步场景中，一个 action 被 dispatch 后可能经过 `acknowledged`（系统确认了请求）再到 `completed`（操作真正完成了）这两个不同的时刻。特别是当 action 是一个长时间运行的外部任务（比如启动一个实验、触发一个数据管道），`acknowledged` 和真正的 `succeeded` 之间可能间隔数小时。如果不区分这两个状态，Capture 阶段在什么时点采集证据就会产生歧义。

**建议：ActionState 应扩展为 `prepared → dispatched → acknowledged → succeeded | effect_unknown | failed`，其中 `acknowledged` 表示外部系统确认受理，`succeeded` 表示操作完成且结果可采集。**

---

**6.8 Idempotency and Side-Effect Contract——非常成熟的设计，几乎无可挑剔**

七层 ID 体系（task_id 到 callback_id）、四级 side-effect 分类（read_only / local_write / external_write / irreversible_submit）、以及五条契约规则，构成了一套完整的幂等性治理框架。特别是"every external action must carry an idempotency key"和"retry policy must respect side-effect class"这两条，直接解决了企业级 Agent 在真实环境中最容易出事故的场景。

**只有一个小建议：** `local_write` 和 `external_write` 的边界在某些场景下可能模糊。比如 Agent 写入一个共享文件系统——对 Agent 来说是"本地"操作，但对企业来说影响范围是"外部"的。建议在定义中补一句边界判定原则，比如"side-effect class is determined by the blast radius of the action, not by the network location of the target"。

---

**6.9 Failure Taxonomy——实用且克制**

八类失败分类覆盖了 TRACE 运行时最常见的失败模式。分类既不过粗（不是只有"success/fail"），也不过细（没有试图穷尽所有可能的错误码）。`contradictory_evidence` 和 `unsafe_action_blocked` 的引入特别好——它们与 Human Gate C 的触发条件和 Harness 的安全治理形成了闭环。

**一个小观察：** 当前分类中缺少一类——**模型层面的失败**。比如 LLM Gateway 返回了不可解析的输出、模型幻觉导致生成了不可执行的 action plan、或者模型拒绝执行（safety refusal）。这些失败不属于现有八类中的任何一类，但在实际运行中一定会遇到。

**建议：增加 `model_output_invalid` 和 `model_refusal` 两类，覆盖 LLM 层面的失败模式。**

### 三、架构级的整体审视——从 V1.0 到 V1.2 的进化评估

经过三轮评审，现在有必要从整体上审视这套架构的成熟度。

**架构的核心赌注是正确的。** TRACE 赌的是：下一代企业 Agent 的竞争力不在单次推理的聪明程度上（那是模型厂商的事），而在持久运行 + 结构化探索 + 可进化 这三者的交叉点上。这个判断我认为是对的。当前的 Agent 框架（无论是 LangChain/CrewAI/AutoGen）都没有认真处理这三个问题，它们仍然在短周期的 prompt engineering 层面竞争。

**从"可进化"这个核心要素来看，V1.2 已经构成了一个可信的闭环。** 具体来说：

进化的数据基础有了——Capture 产出结构化的 trajectory / evidence / failure classification，不再是黑箱。进化的触发机制有了——三种触发模式覆盖了事后总结、批量学习、异常驱动三种时机。进化的隔离机制有了——change_scope 防止归因混乱，champion/challenger 防止盲目更新。进化的知识载体有了——Skill 的五阶段生命周期让经验可以沉淀、验证、推广、退役，而不是在 prompt 里堆砌。进化的成本控制有了——Evolution budget 与 CTS budget 平级对待。

这五个要素共同构成了一个"可信的进化闭环"，而不是一句"系统会越来越好"的空话。

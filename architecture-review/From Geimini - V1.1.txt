我按“严肃架构评审”给结论：

**这版架构有明确的上限，不是空想；但离“可以放心投入核心工程团队实施”还差一层关键硬化。**
它最大的优点，是已经摆脱了“聊天代理”思路，明确把 **Run** 设为持久化主体，把 **Task View 重建、CTS 约束探索、Harness 隔离环境、Evolve 做反馈更新** 放进了核心骨架里。这些判断方向是对的，而且彼此之间基本一致。

我的总评分是：**6.6 / 10**

如果按“是否值得继续投”来判断：**值得继续，但不能直接按当前抽象开大规模实现。必须先补齐几处内核级契约，否则后期会在可进化、恢复、幂等、边界归属上反复返工。**

它已经比大量只会讲 ReAct、工具调用、记忆、技能编排的方案成熟，因为你已经把“长期运行”“上下文重建”“受约束探索”“演化反馈”都提成了一级问题。问题在于：**你现在的“可进化”，更多是一个受控的 postmortem 学习子系统，而不是一个被运行时、版本治理、回滚治理、兼容治理共同托住的系统级能力。**

## 分项打分

### 1. 架构方向正确性：**8.4 / 10**

你抓住了几个真正的一等公民：

* 持久主体是 **Run**，不是 chat thread
* 每次调用都依赖 **Task View 重建**，不是全历史塞 prompt
* 探索在 **CTS** 内进行，而不是无约束 trial-and-error
* 环境访问必须经 **Harness**
* 演化目标是质量与效率，不只是 prompt 调优

这些都不是表面概念，而是能真正改变系统形态的选择。

### 2. 可进化设计：**6.3 / 10**

你已经把 Evolve 从口号变成了有触发、输入、输出、质量护栏、技能晋升路径的子系统，这是明显进步。
但我不会给高分，因为**它还没形成闭环治理能力**。你定义了更新对象，却没有彻底定义：

* 哪些东西允许演化，哪些绝不允许在线改
* 演化单元的版本边界是什么
* 一次演化影响范围如何限制
* 回滚粒度是什么
* 新旧 skill / heuristic / knowledge summary 与旧 run 的兼容关系是什么
* 失败时是禁用单个候选、单个任务族，还是整个策略簇

现在的 Evolve 更像“带安全带的经验沉淀”，还不是“可控、可回滚、可审计、可分层发布的进化系统”。

### 3. 边界清晰度：**6.1 / 10**

文档已经区分了 hi-agent、agent-kernel、agent-core，也区分了 Memory / Knowledge 与外部 Data / IT Systems，这些都对。
但关键边界仍然发虚，尤其是：

* **Task View 重建到底谁负责**
  文档一方面说每次模型调用都要重建 Task View，另一方面 cross-repo sequence 里是 hi-agent 提交 Task View request，agent-kernel 组装 model invocation envelope；kernel requirements 又说 kernel should support bounded Task View assembly inputs。这里所有权没封死。
* **Route Engine 与 Evolution Engine 的分界还不够硬**
  routing heuristic 更新属于 evolve，但 route compare 和 prune 也会依赖被 evolve 更新后的标准。缺少“在线决策面”和“离线进化面”的严格隔离协议。

### 4. 运行时正确性：**5.9 / 10**

你强调 durable runtime、checkpoint、wait/wakeup、watchdog、evidence-based recovery，这说明你知道正确性问题不在“模型聪不聪明”，而在运行时语义。
但当前文档还缺少真正决定正确性的东西：

* Run / Stage / Branch / Action / Wait / Resume / Human Gate 的**形式化状态机**
* 合法状态转移表
* 重复回调、乱序事件、超时回补时的仲裁规则
* 分支合并的语义
* “成功但证据不充分”“失败但产出了可复用局部结果”这类中间态的归类规则

现在你有了词汇表，但还没有“内核真相表”。

### 5. 幂等性与恢复：**5.4 / 10**

这是我最严格扣分的地方。
文档多次提到 checkpoint、resume、evidence-based recovery、trajectory ledger、async callback，但**没有把幂等设计上升为核心契约**。

这会直接出事。因为你的系统不是纯读，它会：

* 调外部系统
* 跑实验
* 生成中间物
* 触发人工门
* 等待异步回调

在这种体系里，如果没有明确的：

* `run_id / stage_id / branch_id / action_id / attempt_id`
* side effect 分类
* 幂等键
* 去重窗口
* 回调幂等消费
* 补偿 / 不可补偿动作标记

那么一旦重试、恢复、跨进程重放，你的“可进化”会首先被“重复执行污染数据”打穿。

### 6. 可实施性：**6.8 / 10**

如果你把 V1 scope 严格限制在文档写的那个研究闭环：检索、分析、写作、打包、人工审阅前交付，那是可以落地的。
但如果你把它当成“企业通用 agent 平台 V1”，那就偏乐观了。真正可实施的前提是：

* 先只做**少数 stage**
* 先只做**少数 harness 能力**
* 先只支持**有限 CTS**
* 先只演化 **routing heuristic + skill candidate**
* 暂时不要让 knowledge updates 和 evaluation baselines 在线互相咬合

否则系统会在“都想做一级能力”时迅速失控。

### 7. 企业治理成熟度：**6.5 / 10**

你已经把 authz、timeout、retry、audit、human gates、feedback、watchdog 放进来了，这是成熟意识。
但还缺三样：

1. **策略发布与回滚治理**
2. **证据与结论的因果链审计**
3. **高风险动作分级授权模型**

没有这三样，企业里只能把它当“高级 copiloting runtime”，还不能当“可信任务执行系统”。

## 这版最值得肯定的地方

### 1. 你真正抓住了“可进化”的前提：**可重建**

可进化不是“会学习”，而是**失败和成功都能被结构化保存，并在下次调用前可重建**。
你把 Task View reconstruction、trajectory capture、durable run 放在核心，这一点非常对。

### 2. CTS 是这版里最有价值的机制

很多架构一谈多路径探索就失控。你至少已经给了：

* Stage Graph
* Trajectory Tree
* budget dimensions
* pruning criteria
* 先 cheap heuristic 再 heavy evaluation 的比较策略

这说明你意识到探索必须被预算化、分层化、可剪枝化。

### 3. 你没有把 Harness 当“工具适配层”

你把它定义成受控执行桥，这很关键。真正长期运行的 agent，必须是：

**LLM 决策，Harness 执行，Runtime 持久化，Ledger 记录，Evolution 改进。**
这个判断是成熟的。

## 我认为最严重的几个问题

## 1. “可进化”还不是系统主轴，只是一个受控后处理面

你现在的 Evolve 有 trigger、有 output、有 guard，但它仍然是**运行后更新**，不是**运行时可治理进化**。

真正的“可进化”架构，至少要回答：

* 进化对象是否分层：skill、route heuristic、knowledge summary、evaluation baseline 是否属于同一发布域
* 进化是否有 blast radius 控制：按 task family、tenant、env、risk level 分域发布没有
* 进化是否有兼容协议：旧 run 恢复时是否强制绑定旧策略版本
* 进化是否有冻结区：哪些核心契约永远不能被在线学习改动

你现在没有把这些写出来，所以“可进化”还偏理想主义。

**严厉一点说：现在更像“可学习”，还不够“可进化”。**

---

## 2. 缺少内核级状态机，导致长程正确性没有落地

你提出了 Run、Stage、Branch、Wait、Wakeup、Recovery、Human Gates，但这些对象之间没有明确状态机。

这会导致几个经典灾难：

* callback 先到还是 timeout 先到，谁赢
* human gate 修改 task contract 后，已打开 branch 是否全部失效
* 某 branch 在 waiting 时，global evolve 更新了 heuristic，恢复后是否沿用旧决策标准
* 某 action 成功但 evidence capture 失败，系统算成功、失败，还是 uncertain

这些不是实现细节，这是**架构真相**。
没有状态机，这个系统后面一定会靠 if-else 堆成泥。

---

## 3. Repository 边界有冲突，未来极易打架

hi-agent 持有 TRACE、CTS、routing、evolution；agent-kernel 持有 durable runtime、LLM Gateway、trajectory ledger；agent-core 提供环境与资源。这个三分思路大体可行。
但从 cross-repository run sequence 看，**Task View 与 model envelope 的装配权责仍可能冲突**。

我给你的判词是：

* **hi-agent 应该拥有“选什么内容”的权力**
* **agent-kernel 应该拥有“如何稳定装配、记录、重放”的权力**
* **agent-core 不应参与任务级认知拼装**

如果这点不钉死，后面上下文工程、检索策略、记忆裁剪、重放复现会在三个 repo 里互相侵蚀。

---

## 4. CTS 机制是亮点，但还不够收敛

CTS 现在仍有“工程爆炸”风险。你定义了预算，但还缺这几个硬机制：

* **branch canonicalization**：两个语义相同但表述不同的分支如何去重
* **failure signature normalization**：什么叫“重复失败签名”
* **branch merge**：分支何时合并、谁保留证据主链
* **value-of-information**：什么时候值得再开新分支，而不是继续补证据

没有这些，CTS 很容易从“受控探索”退化成“昂贵树搜索”。

---

## 5. Skill / Memory / Knowledge 的演化污染风险很高

你区分了 Working/Episodic Memory、Semantic/Procedural Knowledge、Skill，这是对的。
但缺少“从轨迹提炼 skill 时如何避免把任务偶然性写入长期资产”的机制。

例如：

* 成功轨迹里包含某次临时数据路径、某个租户特例、某个一时有效的网页结构
* 这些被误提炼成 Provisional Skill
* 后续跨 run 复用开始污染系统

你现在有三阶段 skill 晋升，但还缺少：

* skill 的适用域声明
* 前置条件声明
* 禁止条件声明
* 依赖的 harness capability / evidence class / model role 声明

没有这些，所谓 skill crystallization 很可能只是“把偶然成功固化成脆弱模板”。

---

## 6. Harness 的治理粒度还不够

你说 Harness 负责 authz、timeout、retry、audit、evidence capture、async callback，这个方向对。
但企业里真正危险的是**副作用治理**，你现在还没写透：

* read / write / mutate / publish / notify / purchase / submit 这些动作是否分级
* 是否支持 dry-run
* 是否支持 approval-required 标记
* 是否支持补偿动作
* 是否区分可重试动作与不可重试动作

不把 side effect taxonomy 写出来，幂等性和安全性都只是抽象好看。

## 我对“可进化”这个核心目标的判断

如果你把“可进化”定义为下面四件事同时成立：

1. 可以替换模型与 provider，核心不塌
2. 可以迭代 route / skill / knowledge / evaluation，而不污染在跑任务
3. 可以从多 run 经验中稳定提升质量与效率
4. 可以对进化造成的回归进行审计、限域、回滚

那么这版架构目前是：

* **1 已经具备不错基础**
* **2 只有半套**
* **3 有结构，但还不够防污染**
* **4 只露出轮廓，没有落到内核契约**

所以我会说：
**它已经把“可进化”从宣传词推进到了架构主题，但还没有推进到可验证的系统能力。**

## 是否合理

**合理，但还不是闭合的。**

合理之处在于，你的主链条是自洽的：

Task Contract → Task View → CTS Route → Harness Act → Capture → Evolve → Next Task View。

不闭合之处在于，链条中最关键的三个“硬连接件”还没有定型：

* 状态机
* 幂等与副作用契约
* 进化版本治理契约

这三个不补，整个架构就会一直停留在“高级设计稿”。

## 是否可实施

**可实施，但必须降维实施。**

我建议你严格按下面顺序落地，否则项目会变形：

### 第一阶段，只做“可运行”

只做：

* 单任务族
* 固定 Stage Graph
* 很小的 CTS budget
* 很少的 Harness capability
* Run / Stage / Branch / Action 的 durable state
* Task View reconstruction
* evidence ledger
* human gate

这一阶段目标不是进化，而是**可恢复、可重放、可审计**。

### 第二阶段，再做“可比较”

补：

* route compare metrics
* failure signature taxonomy
* branch pruning policy
* basic heuristic update

这一阶段目标是**让 CTS 真正产生优于单路径的收益**。

### 第三阶段，最后才做“可进化”

补：

* skill candidate pipeline
* provisional rollout
* task-family-scoped champion/challenger
* regression rollback
* version pinning for old runs

这一阶段才配谈“核心是可进化”。

## 我认为必须马上补的 8 个硬定义

### 1. 统一状态机

至少定义：

`RunState / StageState / BranchState / ActionState / WaitState / ReviewState`

以及所有合法转移。

### 2. 统一幂等键

至少定义：

`task_id, run_id, stage_id, branch_id, action_id, attempt_id, callback_id`

并写清每层幂等语义。

### 3. 统一副作用分类

至少定义：

`read-only / local-write / external-write / irreversible-submit`

没有这个，retry 和 recovery 都是假象。

### 4. 统一证据模型

至少定义：

* evidence ref
* provenance
* confidence
* freshness
* bound_to_branch?
* reusable_across_branches?

### 5. 统一 Task View 装配权责

一句话定死：

* hi-agent 决定内容选择策略
* kernel 负责稳定装配、记录、重放
* core 只提供资源，不做认知拼装

### 6. 统一 Evolution 发布域

明确哪些可按 task family 发布，哪些可按 tenant 发布，哪些只能离线认证后全局生效。

### 7. 统一 Skill 合同

每个 skill 必须带：

* input contract
* preconditions
* forbidden conditions
* evidence requirements
* side-effect class
* version
* rollback policy

### 8. 统一 Failure Taxonomy

比如：

* missing_evidence
* invalid_context
* harness_denied
* callback_timeout
* no_progress
* contradictory_evidence
* unsafe_action_blocked

否则 postmortem 没法进化。

## TRACE V1.1 架构评审意见

### 一、V1.0 遗留问题的回应质量评估

先逐条检查上一轮提出的七个问题，看架构师是否真正解决了。

**问题 1（Evolve 闭环）——实质性改善，但仍有一个盲区**

6.3 节的 Evolve Loop Contract 是 V1.1 中最有价值的增量。触发模式三档分级（per_run_postmortem / batch_evolution / regression_trigger）合理，输入契约的八项数据源覆盖了关键信号，skill crystallization 的三阶段晋级路径（Candidate → Provisional → Certified）是成熟的工程思维，避免了"一次成功就固化"的陷阱。champion/challenger + disable-on-regression 的质量护栏也是对的。

**但有一个盲区没有覆盖：Evolve 对自身的作用范围缺乏隔离。** 当前设计中，Evolve 同时更新四类目标（skill / routing heuristics / knowledge / evaluation baselines）。问题是：如果一次 Evolve 同时更新了路由启发式和评估基线，然后下一次 Run 质量下降了，你无法判断是路由更新导致的退化还是基线漂移导致的误判。这是一个经典的"多变量同时变更导致归因失败"的问题。

**建议：Evolve 的每次执行应该有一个明确的"变更范围声明"（change scope），且同一次 Evolve 不应同时更新评估基线和被评估的对象（routing / skill）。评估基线的更新应该有独立的触发条件和独立的验证周期。**

---

**问题 2（CTS 预算）——解决得干净**

5.3 节的 Budget Model 直接回应了上一轮的核心关切。五个预算维度（分支上限、总分支上限、对比调用上限、对比 token 预算、探索时间预算）覆盖了成本控制的关键面。剪枝条件的五条规则是可操作的。"先用便宜启发式，再用昂贵模型对比"的分层策略也是正确的。

**这一条通过，无进一步意见。**

---

**问题 3（Task View 重建）——方向对了，但缺少一个关键机制**

6.4 节给出了 Task View 的组装优先级（contract → stage → branch → evidence → memory → knowledge → budget）和重建策略（stage-aware、evidence-first、branch-local-first、token 预算约束）。重建验证的三条（记录展示内容、追踪路由质量、事后检测信息缺失）也是对的。

**但缺少一个关键机制：Task View 重建失败时的降级策略。** 在真实场景中，可能出现的情况是：某次重建因 token 预算紧张而丢弃了关键证据，导致模型做出了错误的路由决策。当前设计能在事后检测到这个问题（postmortem 中发现 missing-evidence failure），但没有定义实时的降级行为——比如当 Task View 的组装质量低于某个阈值时，是应该暂停执行请求人工介入（触发 Human Gate），还是应该降级为保守路由（选择已知安全的分支而不探索新分支），还是扩展 token 预算（切换到更大 context window 的模型）？

**建议：补一条 Task View 组装的"完整性检查"规则。当必要信息（比如 must-keep evidence）因预算不足无法全部纳入时，系统应有明确的降级行为定义，而不是静默裁剪后继续执行。**

---

**问题 4（三仓库边界）——显著改善**

6.6 节的 Cross-Repository Run Sequence 是 V1.1 中第二个高价值增量。七步序列清晰地展示了 hi-agent → agent-kernel → agent-core 的交互流程，特别是明确了：hi-agent 拥有语义判断权（路由质量、效率评估、Evolve 触发），agent-kernel 拥有持久化和编排权（run state、trajectory ledger、LLM Gateway），agent-core 拥有环境能力供给权。

**但序列图暴露了一个新问题：步骤 3 中"agent-kernel assembles model invocation envelope through LLM Gateway contract"。** 这意味着 kernel 需要组装 prompt——但 prompt 的内容语义（该放什么 evidence、该用什么 system prompt 模板）是 hi-agent 层的策略决策。如果 kernel 只是透传 hi-agent 给定的完整 prompt，那 kernel 的"assemble"职责是什么？如果 kernel 有自己的组装逻辑，那 hi-agent 对 Task View 的控制权就被分割了。

**建议：明确"model invocation envelope"的组装责任分界。推荐方案是 hi-agent 生成完整的 Task View 内容（语义层），kernel 只负责封装为具体 provider 的 API 格式（传输层）。"Assemble"一词在当前文档中有歧义，应该替换为更精确的表述。**

---

**问题 5（LLM Gateway 角色粒度）——完全采纳，执行干净**

公开能力角色从 7 个细粒度缩减为 3 个粗粒度（heavy_reasoning / light_processing / evaluation），细粒度角色下沉为 hi-agent 内部路由策略。11.2 节的表述"Fine-grained roles ... should remain upper-layer routing policy, not kernel public API"精准。

**这一条通过，无进一步意见。**

---

**问题 6（人类参与）——实质性改善**

6.5 节定义了四个 Human Gate（合同修正 / 路由选择 / 中间产物编辑 / 最终审批），且明确了人类介入可以修改合同、剪枝分支、注入证据、编辑产物。这比 V1.0 的被动模型好得多。

**但有一个实施层面的隐患：Human Gate 触发的条件没有定义。** Gate A-D 是四种"可以发生"的介入类型，但什么条件下系统主动触发它们？是由 CTS 预算耗尽触发？由质量指标低于阈值触发？由任务被标记为高风险触发？还是纯粹由人类主动发起？如果只是被动等待人类主动发起，那在 7×24 长任务中，人类不在线时这些 Gate 永远不会被触发，系统可能在错误的方向上消耗大量资源。

**建议：为每个 Human Gate 定义至少一个"系统主动触发条件"。比如 Gate B 应在"探索预算消耗超过 60% 但尚无可行分支"时主动触发；Gate C 应在"中间产物的模型自评低于阈值"时触发。**

---

**问题 7（验证策略领域绑定风险）——未回应**

V1.1 没有对这条做出任何调整。文档仍然只有科研场景作为唯一验证域，没有影子验证计划。

**判定：不阻塞架构审批，但维持原建议——在验证阶段至少用一个非科研场景做影子验证。**

### 二、V1.1 新增内容引入的新问题

除了上一轮遗留问题，V1.1 的新增内容引入了两个需要注意的新问题。

**新问题 A：Skill 系统的版本管理和冲突解决未定义**

Skill 的三阶段晋级路径定义了"怎么晋级"，但没有定义"怎么退役"和"怎么处理冲突"。在真实系统中会出现的情况包括：一个 Certified Skill 因模型能力升级而变得次优（比如旧 skill 是为弱模型设计的 workaround，强模型不需要了）；两个 Provisional Skill 覆盖相同的任务阶段但给出不同的策略建议，路由引擎如何选择？Skill 的版本迭代和旧版本的淘汰机制是什么？

**建议：Skill 需要一个生命周期模型（creation → provisional → certified → deprecated → retired），以及冲突消解的优先级规则。**

---

**新问题 B：Evolve 的计算开销和触发频率缺乏预算约束**

CTS 有了明确的预算模型，但 Evolve 本身没有。`batch_evolution` 跨多个 Run 汇总分析，这本身是一个昂贵的操作（需要读取多个 Run 的 trajectory summary，可能需要多次 LLM 调用来提取模式）。`regression_trigger` 如果阈值设得太敏感，可能在正常波动中频繁触发。这些开销如果不受控，Evolve 本身就会成为成本黑洞。

**建议：为 Evolve 定义自己的预算约束，包括 batch_evolution 的最大分析 Run 数量、单次 Evolve 的最大 LLM 调用次数、regression_trigger 的最小冷却间隔。**

### 四、最终建议

**架构方向确认通过。V1.1 已经达到了可以进入实现规划的成熟度。** 但在冻结词汇表之前，建议完成以下四项补充（按优先级排序）：

1. **Evolve 变更隔离规则**——定义 Evolve 的变更范围声明机制，禁止同一次 Evolve 同时更新评估基线和被评估对象。这是保证进化闭环可信赖的底线。

2. **Task View 组装完整性检查与降级策略**——当必要信息无法全部纳入时，定义明确的降级行为（暂停 / 保守路由 / 升级模型），而不是静默裁剪。

3. **Human Gate 的系统主动触发条件**——每个 Gate 至少定义一个系统侧的触发阈值，避免长任务中人类不在线时系统在错误方向上空转。

4. **Evolve 自身的预算约束**——给 Evolve 加上与 CTS 同级别的成本控制，防止进化过程本身成为开销黑洞。

这四项都不需要改变架构的核心结构，只需要在现有框架内补充具体的规则定义。完成后，可以冻结 TRACE 词汇表，启动 hi-agent 到 agent-kernel 的契约映射。

架构师做得不错——V1.1 的质量提升是实质性的，不是表面修补。

## TRACE 架构评审意见

### 一、整体判断

这是一份**高于平均水准的架构设计文档**。概念体系清晰，约束条件明确，分层合理，且刻意回避了当前 Agent 社区中常见的"大而全但落不了地"的毛病。能看出作者有真实的工程经验，而不是在纸上画概念图。

但作为严格评审，我需要指出**七个核心问题**，按严重程度排序。

---

### 二、需要认真回答的七个问题

**问题 1：Evolve 闭环的驱动信号定义不足——这是最大的短板**

你把"可进化"定位为架构的核心要素，但文档中 Evolve 环节恰恰是定义最薄弱的部分。目前只列了四个动作（update skill candidates / knowledge summaries / routing heuristics / evaluation baselines），但没有回答关键问题：

- 谁触发 Evolve？是每次 Run 结束自动触发，还是攒批触发，还是由质量指标跌破阈值触发？
- Evolve 的输入契约是什么？Capture 产出的哪些结构化数据会流入 Evolve？格式是什么？
- Skill crystallization 的判定条件是什么？一条轨迹成功一次就结晶，还是要跨多次 Run 验证稳定性？
- Routing heuristics 的更新是规则驱动还是模型驱动？如果是模型驱动，谁来评估更新后的路由质量没有退化？

**判定：如果 Evolve 没有明确的触发机制、输入契约和质量护栏，那它就不是架构组件，而是一个愿望清单。** 作为你自己定义的核心差异点，这里必须补齐，否则 TRACE 和"ReAct + 持久化"没有本质区别。

---

**问题 2：CTS 的运行时成本和复杂度未被正视**

CTS（Constrained Trajectory Space）是架构中最有野心的设计，但文档完全没有讨论其代价：

- **分支爆炸控制**：Stage Graph 有 5 个阶段，每个阶段允许多条分支。即使每阶段只开 3 条分支，组合空间也会迅速膨胀。文档说了"prune"，但没有定义剪枝策略——是基于成本预算、时间预算、还是质量下界？
- **分支对比的成本**：`compare_routes` 本身需要调用 LLM，而多条分支并行意味着对比开销是 O(n²)。在企业场景中，这个成本是否可接受？
- **Stage Graph 的来源**：5 阶段的 Stage Graph 目前是硬编码在文档中的。不同任务类型的 Stage Graph 是否不同？谁来定义它？如果由模型来推断，这本身就是一个高风险的决策点。

**判定：CTS 需要一个明确的"探索预算"机制，否则在真实企业场景中要么退化为单线执行（没人敢开分支），要么失控。**

---

**问题 3：Context OS / Task View 的重建策略是隐藏的技术风险**

文档反复强调"上下文有界、需要重建"，这个约束识别得很好。但 Task View 的重建恰恰是整个架构中实现难度最高的部分，而文档几乎没有展开：

- 重建时的信息选择策略是什么？是按时间衰减、按相关性排序、还是按阶段裁剪？
- 重建质量如何验证？如果某次重建丢失了关键信息导致路由错误，系统如何检测？
- 重建本身消耗的 token 预算如何管理？在一个已经运行了数小时、积累了大量 trajectory 数据的 Run 中，光是重建上下文可能就要消耗相当比例的 context window。

**判定：这不是"以后实现时再细化"的问题。Task View 重建策略的选择会直接影响上层 Route 和 Evolve 的行为，是架构级决策。**

---

**问题 4：三仓库拆分的边界在关键场景下会模糊**

`hi-agent` / `agent-kernel` / `agent-core` 的三仓库拆分思路是合理的——上层策略、中间运行时、下层能力供给。但有几个场景下边界会出问题：

- **Trajectory Ledger** 放在 kernel 中，但 trajectory 的语义结构（什么是"成功"、什么是"高效"）由 hi-agent 定义。这意味着 kernel 要存储它不理解语义的数据，而 hi-agent 要查询它不拥有的存储。这个交互契约的复杂度容易被低估。
- **Harness** 的定义横跨 kernel 和 core：kernel 提供执行原语，core 提供具体环境接入。但 harness 的 retry/timeout 策略是业务相关的（比如"调用实验平台"和"调用代码编译器"的重试策略完全不同），这部分归谁？
- 如果三个仓库的开发节奏不一致，契约变更的协调成本可能会很高。

**判定：建议在文档中补一张"跨仓库调用序列图"，把至少一个完整的 Run 生命周期的跨仓库交互画清楚。抽象层次对不对，要靠具体场景来验证。**

---

**问题 5：LLM Gateway 的"能力角色"抽象有过度设计风险**

将 LLM 调用抽象为 `reason / judge / summarize / extract / write / review / compare_routes` 这样的能力角色，想法是好的，但有两个隐患：

- **角色边界不清晰**：`reason` 和 `judge` 的区别是什么？在实际实现中，很多任务需要同时推理和判断，强行拆分角色反而增加了调用次数和延迟。
- **角色到模型的映射不稳定**：今天你可能认为 `summarize` 用轻量模型就够了，但如果摘要质量成为瓶颈，就需要换强模型。这种映射变更的频率可能远高于预期，导致 Gateway 层变成一个需要频繁调整的配置中心。

**判定：建议先从 2-3 个粗粒度角色开始（比如 heavy-thinking / light-processing / evaluation），而不是一上来就定义 7 个细粒度角色。细粒度的分类可以作为路由策略的内部逻辑，不必暴露为顶层契约。**

---

**问题 6：人类参与模型过于被动**

文档中人类的角色只出现在两个地方：feedback signals 作为输入、human review 作为等待条件。但在企业场景中，人类的参与模式远比这复杂：

- 人类可能需要在 Route 阶段介入，选择探索方向（而不只是事后评审）。
- 人类可能需要修正 Task Contract（任务做到一半发现目标变了）。
- 人类可能需要直接编辑中间产物，而不是只给 feedback。

当前架构对这些场景没有预留位置。如果后期强行加入，可能会破坏 CTS 的一致性。

**判定：建议在 Run 的生命周期中定义 2-3 个明确的"人类介入点"（Human Gate），并定义介入后对 CTS 状态的影响。**

---

**问题 7：验证策略的风险——科研场景可能误导架构决策**

选择科研任务作为第一个验证域的理由（长周期、多阶段、强证据）是合理的。但科研场景有一些特殊性，可能导致架构被带偏：

- 科研任务的 Stage Graph 相对固定（文献综述→实验→分析→写作），但企业任务的阶段结构变化很大。如果 Stage Graph 在验证阶段被隐式固化，迁移到其他领域时可能发现它不够灵活。
- 科研任务的"成功"有相对清晰的定义（论文可提交），但很多企业任务的成功标准是模糊的、多维的、甚至是政治性的。Evolve 基于的质量信号在这种场景下可能严重不足。

**判定：不阻塞，但建议在验证阶段有意识地用至少一个非科研场景（比如一个典型的企业数据分析任务）做影子验证，确保架构假设不会被单一领域绑架。**

---

### 三、架构优点（值得保留的设计决策）

公平起见，以下设计点是经过深思熟虑的，应该坚持：

1. **"Run 是持久主体，不是对话线程"**——这一点比目前 90% 的 Agent 框架认知更成熟。它直接决定了架构能否支撑真正的长任务。

2. **Context 有界、需要重建的约束识别**——大多数 Agent 框架回避这个问题，假装 context window 会无限增长。你直面了它。

3. **Harness 作为必选架构组件**——不让模型直接操作环境，强制走 harness，这在企业场景中不是可选项，是必须项。安全、审计、重试、证据采集都依赖它。

4. **Memory 和 Knowledge 的明确区分，以及它们与 Data Systems / IT Systems 的边界**——这四个概念在很多架构中被混为一谈。你的分类是干净的。

5. **三仓库拆分的意图**——虽然边界细节需要打磨，但"策略层不要和运行时耦合"的分离意图是正确的。
